## 2. 為什麼只跑 `test/test.sh` 不夠？

作業說明指出：

> 繳交後實際計算作業成績時**會修改變數的數值**，例如
> `吾有一爻。曰陰。名之曰「陰陽」。` 可能改成 `吾有一爻。曰陽。名之曰「陰陽」。`；
> `submit_test_hw2` 指令**不會**改變數數值。

`test/test.sh` 的判分方式，是把編譯/執行結果與**固定的** `*.verbose` / `*.expected`
參考檔逐字比對。這些參考檔是依「原始數值」產生的。也就是說：

- 通過 `test.sh` 只證明「**在這一組數值下**輸出正確」。
- 助教改了數值後，參考檔會以**助教的標準解**重新產生；若你的編譯器其實是
  巧合對上某組數值、而非真的在運算，換了數值就會失分。

因此本報告額外設計了 **`test/validate.sh`（數值穩健性驗證）**，
直接模擬「助教改數值」的情境。

## 3. `test/validate.sh` 的方法

核心想法：**不依賴任何固定參考檔**。對每一種語言特性，
用「多組不同數值」即時產生 `.wy` 原始碼，
再用 **bash 獨立算出應有輸出**（與編譯器邏輯互不相干），
最後 `wyc` 編譯 + 執行，比對結果。

若兩者一致，代表編譯器是**真的在運算**，而非背參考答案 —— 這正是改數值後仍會通過的保證。

- 數字字面值由腳本內 `num2hanzi()` 依編譯器接受之標準漢字寫法產生（例：`10 → 一十`、`1013 → 一千零一十三`）。
- 期望值以 bash 算術（`$(( ))`）算出；整數除法/取餘的截斷方向與 C 一致，故與編譯器相符。
- 每個 `.wy` 都編譯為實際可執行檔後執行，輸出再以「去尾端空白行」正規化後比對（與 `test.sh` 的 `normalize` 同義）。

### 涵蓋的特性（48 案例）

| 群組 | 特性 | 掃描的數值 / 變化 |
|------|------|-------------------|
| **T1** 數字往返 | 漢字數字 → `書之` 須等於原值（詞法轉數） | 0,1,2,7,9,10,13,20,50,99,100,101,131,200,1000,1013,9999 |
| **T2** 爻 (bool) | `曰陰`→印「陰」、`曰陽`→印「陽」 | 陰 / 陽（助教典型改動） |
| **T3** 字串 | `言` 原樣輸出 | 中文、標點、ASCII 混合 4 組 |
| **T4** 算術 | 加 / 減 / 乘 / 除 / 取餘（所餘幾何） | 7 組 (甲,乙)：含整除、餘數、相等 |
| **T5** 決策 | `若 / 或若 / 若非` 依數值命中分支 | 5 組，分別命中 大 / 等 / 小 |
| **T6** 迴圈 | `為是 n 遍` 計次；`恆為是 … 乃止` 倒數 | n = 1,2,3,5 |
| **T7** 字串相加 | `加 甲 以 乙` = 串接 | 3 組 |
| **T8** 陣列 | `充`、`之長`、1-based 索引 `之 idx` | 長度 3/4/5，取頭/中/尾 |

### 執行輸出（節錄）

```
══ 數值穩健性驗證 (value-robustness) ══

T4 算術：加/減/乘/除/餘，掃描多組數值
  PASS  算 甲=10 乙=3 (和13 差7 積30 商3 餘1)
  PASS  算 甲=50 乙=7 (和57 差43 積350 商7 餘1)
  ...
T5 決策：若/或若/若非，依數值命中正確分支
  PASS  決策 甲=5 乙=3 → 大
  PASS  決策 甲=3 乙=3 → 等
  PASS  決策 甲=2 乙=4 → 小
  ...
══ 穩健性結果 ══
  通過 PASS: 48    失敗 FAIL: 0
全數通過：改變數值後編譯器仍輸出正確結果。
```
